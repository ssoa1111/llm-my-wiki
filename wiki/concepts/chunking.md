# 청킹 (Chunking)

> RAG에서 문서를 검색 가능한 단위로 분할하는 전략. 청킹 품질이 RAG 답변 품질을 직접적으로 결정한다.

## 핵심 내용

[RAG](../concepts/rag.md)는 문서를 "청크(chunk)"라는 단위로 잘라 [벡터 데이터베이스](../tech/ai/vector-database.md)에 저장한다. 어떻게 자르느냐가 검색 정확도와 답변 품질을 좌우한다.

### 청킹 방식 5가지

**1. 고정 크기 (Fixed-size)**
- 500자씩 무조건 분할, 겹침(overlap) 50자
- 속도: 빠름 / 품질: 낮음 / 사용: 10%
- 단점: 문맥이 중간에 끊길 수 있음

**2. 재귀적 분할 (Recursive) ⭐ 추천**
- 우선순위 순서로 분할: 문단(\\n\\n) → 문장(\\n) → 단어 → 글자
- 속도: 중간 / 품질: 높음 / 사용: 70%
- 문맥 보존이 가장 우수한 범용 방식

**3. 의미 기반 (Semantic)**
- LLM이 주제가 바뀌는 지점을 찾아서 분할
- 속도: 느림 / 품질: 최고 / 사용: 15%
- 비용이 비싸지만 의미 단위로 완벽하게 분할

**4. 문서 구조 기반**
- 마크다운 헤더 단위, 코드는 함수/클래스 단위로 분할
- 구조가 명확한 문서에 적합

**5. 문장 기반**
- 3-5문장씩 묶기
- 자연스럽지만 청크가 너무 작을 수 있음

---

### 청킹-임베딩 매핑 방식

기본적으로 1청크 = 1임베딩이지만, 더 정교한 방식들이 있다.

**Parent-Child ⭐ 추천**
- 작은 청크로 검색 → 큰 청크(부모)를 컨텍스트로 반환
- 구현: 보통 / 정확도: 85% / 사용: 15%
- 검색 정확도 + 풍부한 컨텍스트를 동시에 달성

**Multi-vector**
- 청크 1개에서 여러 관점의 임베딩 생성 (원본, 요약, 예상 질문 등)
- 구현: 어려움 / 정확도: 95% / 비용: 3-5배
- 의료·법률처럼 극한 정확도가 필요할 때

**계층적 (Hierarchical)**
- 요약 임베딩(개요)과 원본 임베딩(상세) 두 개 저장
- 1차 요약으로 필터링 후 2차 원본으로 정밀 검색

### 실무 선택 가이드

```
대부분의 케이스   → 재귀적 분할 + 1:1 임베딩 (빠르고 충분히 좋음)
품질 개선 필요   → Parent-Child 추가 (큰 노력 없이 정확도 ↑)
극한 정확도 필요  → Multi-vector (의료/법률)
```
