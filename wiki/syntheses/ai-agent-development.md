# 신뢰할 수 있는 AI 에이전트 개발 — 프롬프트 설계부터 환각 방지까지

> CoT/Few-shot/ReAct 프롬프트로 사고 과정을 유도하고, 고급 RAG로 검색 정확도를 높이고, 단계 분리와 출력 검증으로 환각을 방지하며, Claude Skill 8단계로 에이전트를 고도화하는 신뢰성 높은 AI 에이전트 개발 전략.

## 핵심 내용

신뢰할 수 있는 AI 에이전트는 네 가지 레이어가 서로 맞물려 동작한다: **프롬프트 설계 → 검색 전략 → 출력 검증 → 에이전트 고도화**. 각 레이어가 다음 레이어의 품질을 결정하며, 피드백 사이클로 연결된다.

---

### 1단계: 프롬프트 설계 — 사고 과정 유도

에이전트가 신뢰할 수 있는 출력을 생성하려면, 단순한 지시보다 **사고 구조를 강제하는 프롬프트**가 필요하다.

| 기법 | 역할 | 에이전트 적용 |
|------|------|--------------|
| **Chain-of-Thought (CoT)** | 추론 과정을 단계별로 출력 | 복잡한 판단 전 근거 생성 강제 |
| **Few-shot** | 예시로 패턴 유도 | 일관된 출력 형식 고정 |
| **ReAct** | 추론과 행동을 교차 실행 | 검색 필요 여부를 스스로 판단 |
| **Self-Consistency** | 여러 번 생성 후 다수결 채택 | 정답 안정성 향상 |
| **Structured Output** | JSON/표 형식 지정 | 검증 자동화를 위한 파싱 가능 출력 |

**핵심 원칙**: 프롬프트에 근거 의무화 규칙을 명시한다. "모든 주장은 반드시 입력 데이터에서 근거를 제시하라. 출처가 없으면 '근거 없음'으로 표시하라." 이 규칙 하나가 환각을 사전에 차단하는 첫 번째 방어선이다.

**에이전트용 구조적 프롬프트 템플릿**:
```
### [역할]
페르소나 + 대상 독자 + 핵심 목표

### [입력 정보]
- 주요 입력 변수
- 제약사항 (정보 부족 시 만들어내지 말 것)
- 우선순위

### [절차]
1. 입력 검증 및 초기 판단
2. ReAct 루프: 추론 → 필요 시 검색 → 재추론
3. 자기 검증 및 출력 형식화

### [출력 형식]
- 최종 출력 형식 (JSON/표/자연어)
- 정보 부족 시 대처법
```

### 2단계: 검색 전략 — 고급 RAG로 정확도 향상

프롬프트가 사고 구조를 잡는다면, RAG는 사고의 재료(근거 문서)를 공급한다. 기본 RAG는 질문 유형과 무관하게 항상 같은 방식으로 검색하고, 검색 실패 시 그냥 진행하는 문제가 있다. 고급 RAG 패턴이 이를 해결한다.

**Adaptive RAG — 질문 유형에 따라 경로 분기**
- 내부 지식 질문 → 벡터 DB 검색
- 최신 정보 질문 → 웹 검색
- 간단한 질문 → 직접 답변 (불필요한 검색 생략)

**Corrective RAG — 검색 품질 평가 후 Fallback**
- 검색 결과를 LLM으로 관련성 평가 (0-100점)
- 점수 ≥ 70 → 답변 생성 / 점수 < 70 → 웹 검색으로 재시도
- 오래된 정보나 저품질 문서를 자동 필터링

**Self-RAG — 생성된 답변 자체를 검증**
- 답변 생성 후 3가지를 병렬 검증: ①문서 근거 여부 ②환각 여부 ③질문 관련성
- 검증 실패 시 재생성 (최대 3회)
- 환각률을 20-30%에서 5-10% 이하로 감소

**세 패턴 결합 — Ultra-RAG 파이프라인**:
```
[Adaptive] 질문 분류 → 소스 선택
    ↓
[Corrective] 관련성 평가 → 필요 시 웹 검색
    ↓
[Self-RAG] 답변 검증 → 필요 시 재생성
    ↓
최종 출력 (환각률 <5%)
```

구현은 LangGraph로: 순환(retry loop), 조건부 분기, 상태 관리가 필요하기 때문에 LangChain의 선형 체인으로는 구현하기 어렵다.

### 3단계: 출력 검증 — 생성 단계 분리와 사실 검증

검색이 좋은 재료를 공급해도, 생성 단계에서 환각이 발생할 수 있다. 핵심 전략은 **한 번에 다 시키지 않는 것**이다.

**생성 단계 분리 패턴**:
```
1단계 (사실 추출): 원본 데이터에서 수치/사실만 JSON으로 추출. 해석 금지.
2단계 (해석):     JSON 기반 인사이트만 정리. 반드시 수치 포함, 데이터 범위 안에서만.
3단계 (보고서):   인사이트를 보고서 형식으로 재구성.
```

단계를 나누면 각 단계의 출력이 다음 단계의 입력이 되어 오류 탐지가 쉬워진다. 1단계에서 잘못된 수치가 나오면 2단계 이전에 잡을 수 있다.

**출력 검증 레이어**:
- **사실 검증**: 결과물을 2차 모델로 "이 내용이 입력 근거와 일치하는지" 검증
- **일관성 체크**: 수치, 날짜, 이름을 정규식으로 추출해 원본 데이터와 매칭
- **교차 질문**: 같은 질문을 다르게 여러 번 물어보고 불일치 시 경고

**프롬프트→검색→생성→검증 피드백 사이클**:
```
[프롬프트: 근거 의무화]
        ↓
[검색: Adaptive/Corrective RAG]
        ↓
[생성: 단계 분리 + 구조화 출력]
        ↓
[검증: Self-RAG 자동 검증 + 사실 검증 레이어]
        ↓
    검증 통과? → 최종 출력
    검증 실패? → 재생성 루프 (최대 3회)
        ↑________________↓
```

### 4단계: 에이전트 고도화 — Claude Skill 8단계

신뢰성 있는 에이전트를 시스템으로 만들려면, 위의 전략들을 잘 정의된 Skill로 캡슐화해야 한다.

**에이전트의 도구 선택 원리**: 에이전트는 도구의 `description`만 보고 선택한다. description이 명확할수록 올바른 도구가 호출된다. 따라서 환각 방지 파이프라인이나 RAG 검색 로직도 명확한 description을 가진 Skill로 만들어야 한다.

**8단계 Skill 고도화 워크플로우**:

| 단계 | 목표 | 핵심 질문 |
|------|------|----------|
| 1. 초안 생성 | 기본 개념과 워크플로우 | "이 스킬이 무엇을 하는가?" |
| 2. 정량화 | 모호한 기준을 체크리스트/점수로 | "점수 기준이 명확한가?" |
| 3. 분기 로직 | 경계선 케이스 5가지 의사결정 트리 | "엣지케이스를 처리할 수 있는가?" |
| 4. 프로그래밍 사고 | 입력/조건/반환값 명확화 | "파이썬 함수처럼 표현할 수 있는가?" |
| 5. 실전 패턴 | 대표 입력 10개 → 공통 패턴 추출 | "실제 데이터에 작동하는가?" |
| 6. 메타 확장 | 스킬이 자기 자신에게도 적용되는가 | "재귀적으로 개선 가능한가?" |
| 7. 예시 강화 | Before/After 비교 예시 3개 | "기대 결과가 명확한가?" |
| 8. 최종 다듬기 | YAML 메타데이터 + 구조화 마크다운 | "문서가 완결성 있는가?" |

**좋은 Skill의 기준**: 입력/출력 명확, 판단 기준 수치화, 엣지케이스 처리 포함, Before/After 예시, 의사결정 트리 구조화. 이 기준은 프롬프트 설계 원칙(구조적 프롬프트 템플릿)과 일치한다 — Skill 고도화는 곧 프롬프트 고도화다.

---

### 전략 통합: 신뢰성 매트릭스

| 레이어 | 핵심 기법 | 환각 방지 효과 |
|--------|----------|--------------|
| 프롬프트 설계 | CoT + ReAct + 근거 의무화 | 선제적 제어 |
| 검색 전략 | Adaptive + Corrective RAG | 근거 품질 보장 |
| 출력 검증 | Self-RAG + 단계 분리 | 생성 후 자동 교정 |
| 에이전트 고도화 | Skill 8단계 | 시스템 수준 신뢰성 |

**성능 기준점**: 기본 RAG(환각률 20-30%) → Ultra-RAG + 검증 레이어(환각률 <5%). 프로덕션 AI 에이전트는 이 네 레이어가 모두 작동해야 한다.
