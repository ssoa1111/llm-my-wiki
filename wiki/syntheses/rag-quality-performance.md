# RAG 품질 & 성능 최적화 실전 가이드

> 벡터 DB 선택, 유사도 측정, 검색 정확성 향상, 속도 최적화를 단계별로 연결하는 실전 RAG 구축 가이드

## 핵심 내용

### 전체 최적화 로드맵

```
1단계: 인프라 선택 (벡터 DB + 유사도 방식)
       ↓
2단계: 검색 정확성 확보 (Multi-Step + Confidence)
       ↓
3단계: 성능 최적화 (병렬처리 + 캐싱 + 경량 모델)
       ↓
4단계: 운영 패턴 확립 (점진적 고도화 전략)
```

---

### 1단계: 벡터 DB 선택 — 규모별 최적 선택

전용 벡터 DB가 pgvector보다 빠른 이유는 알고리즘, 메모리 상주, 언어 최적화 세 가지에 있다.

| 알고리즘 | 설명 | 성능 |
|---------|------|------|
| IVFFlat (pgvector) | 모든 벡터를 순차 비교 | 100-500ms |
| HNSW (Qdrant, Pinecone) | 그래프로 빠르게 탐색 | 10-60ms |

**규모별 선택 기준 (문서 100만 개 기준 속도 비교)**

| DB | 속도 | 비용 | 적합한 규모 |
|----|------|------|------------|
| pgvector | 100-500ms | 무료 | 문서 < 10만 개 |
| Qdrant | 20-60ms (~5배) | 무료 1GB | 문서 10-100만 개 |
| Milvus | 15-50ms (~8배) | 무료 (자체 호스팅) | 극한 성능 필요 시 |
| Pinecone | 10-50ms (~10배) | $70/월~ | 문서 100만 개+, 관리 편의 |

> 실무 권고: 대부분 개인/소규모 프로젝트는 Supabase Vector(pgvector)로 시작하고, 성능 문제 발생 시 Qdrant로 마이그레이션.

---

### 2단계: 유사도 측정 방식 — 올바른 메트릭 선택

벡터 DB 내 검색은 쿼리 벡터와 저장 벡터 간 유사도 계산으로 동작한다.

| 방식 | 범위 | 특징 | 사용 비율 |
|------|------|------|----------|
| 코사인 유사도 | -1 ~ 1 | 벡터 방향(의미)만 비교, 길이 무관 | **90%** — RAG 표준 |
| 유클리드 거리 | 0 ~ ∞ | 벡터 크기까지 고려, 절대 차이 반영 | 8% — 이미지 임베딩 |
| 내적 | -∞ ~ ∞ | 정규화 시 코사인과 동일, 속도 빠름 | 2% — 추천 시스템 |

```
RAG/텍스트 검색 → 코사인 유사도 (OpenAI, Cohere 모두 권장)
이미지 유사도  → 유클리드 거리
추천 시스템    → 내적 (속도 우선)
```

**직관적 이해**: 벡터의 방향(의미)이 같다면 길이(문서 길이)가 달라도 같은 개념이므로, 텍스트 의미 비교에는 코사인이 최적.

---

### 3단계: 검색 정확성 향상 — Multi-Step + Confidence 전략

단순 벡터 검색은 쿼리 표현 방식에 따라 결과가 크게 달라진다. 아래 7가지 전략으로 AI 에이전트가 능동적으로 검색 품질을 개선한다.

**Confidence 레벨 시스템 (임계값 기반 분기)**

| Confidence | 유사도 | 대응 전략 |
|------------|--------|----------|
| high | ≥ 0.7 | 바로 결과 활용 |
| medium | ≥ 0.5 | 결과 + 대안 확인 |
| low | ≥ 0.3 | 대안 확인 또는 재검색 |
| none | < 0.3 | 키워드 변경 후 재검색 |

**Multi-Step 검색 4단계**

```
1. 쿼리 최적화
   긴 문장 → 핵심 키워드 + 동의어 고려
   ❌ "URL을 어떻게 구성해야 할까요?"
   ✅ "URL 구조", "메타태그"

2. 결과 해석
   Confidence 레벨 기반 판단

3. 결과 불충분 시 복구
   전략 A: 키워드 변형 (일반화, 동의어, 언어 변환)
   전략 B: 전체 주제 목록 조회 후 선택
   전략 C: alternativeTopics에서 재선택

4. 응답 생성
   출처 명시 + 신뢰도 표시
```

**검색 전략별 정확도/속도 비교**

| 전략 | 정확도 | 속도 | 비용 | 추천 상황 |
|------|--------|------|------|----------|
| 벡터만 | 70% | 0.1초 | 저렴 | 프로토타입 |
| 하이브리드 (벡터 + BM25) | 85% | 0.15초 | 저렴 | 대부분 경우 (권장) |
| 하이브리드 + MMR + 재순위 | 90% | 0.5초 | 비쌈 | 의료/법률/금융 |

> 핵심 인사이트: 임계값을 0.5 → 0.3으로 낮춰 False Negative(관련 결과 누락)를 방지하고, Confidence 시스템으로 품질을 관리하는 것이 핵심. 도구 호출 횟수는 최대 5회로 제한해 무한 루프와 토큰 비용을 통제한다.

---

### 4단계: 속도 60% 개선 — 6가지 최적화 기법

Ultra-RAG의 10-15초를 4-6초로 줄이는 전략.

**기법별 단축 효과**

| 기법 | 개선 효과 | 설명 |
|------|----------|------|
| 병렬 처리 | -4초 (50%) | `asyncio.gather`로 DB 검색 + 웹 검색 동시 실행 |
| 빠른 모델 전략 | -2초 | 라우팅/평가 → GPT-3.5, 최종 답변 → GPT-4 |
| 캐싱 | -3초 (재방문) | 임베딩 캐싱 + 검색 결과 캐싱 (TTL 30-60분) |
| 검증 단계 통합 | -4초 | 3번 개별 LLM 호출 → 1번에 일괄 처리 |
| 스트리밍 | 체감 즉시 | `llm.astream()`으로 생성하면서 실시간 전송 |
| 재시도 횟수 제한 | -8초 | max_retries 3회 → 1회 |

**캐싱 전략 상세**

```python
# 1. 임베딩 캐싱 (동일 쿼리 반복 시 재계산 방지)
embedding_cache = {}
def get_embedding(text):
    if text not in embedding_cache:
        embedding_cache[text] = embed(text)
    return embedding_cache[text]

# 2. 검색 결과 캐싱 (TTL 30분)
@cache(ttl=1800)
async def cached_search(query_hash):
    return await vectorstore.search(query_hash)
```

### 점진적 고도화 전략 (오버엔지니어링 방지)

```
80% 케이스: 기본 RAG         (3초, $0.01/회)   ← 시작점
15% 케이스: + Adaptive RAG   (4초, $0.015/회)  ← 문제 생기면 추가
 5% 케이스: + Self-RAG       (의료/법률/금융)   ← 환각 심할 때만
```

**실무 권장 진화 경로**

```
시작: pgvector + 기본 벡터 검색 + 코사인 유사도
  ↓ 정확도 부족 시
하이브리드 검색(벡터 + BM25) + Confidence 레벨 적용
  ↓ 속도 문제 시
병렬 처리 + 캐싱 + 경량 모델 전략 적용
  ↓ 규모 확장 시
pgvector → Qdrant 마이그레이션
  ↓ 극한 정확도 필요 시
재순위화(Cohere Rerank) + Self-RAG + Qdrant/Pinecone
```

### 의사결정 요약표

| 요소 | 소규모 (< 10만 문서) | 중규모 (10-100만) | 대규모 (100만+) |
|------|--------------------|--------------------|----------------|
| 벡터 DB | pgvector (Supabase) | Qdrant | Pinecone / Milvus |
| 유사도 | 코사인 | 코사인 | 코사인 |
| 검색 전략 | 벡터 단독 | 하이브리드 | 하이브리드 + 재순위 |
| 속도 최적화 | 캐싱 | 병렬 처리 + 캐싱 | 전체 6가지 기법 |
| 목표 응답시간 | 1-3초 | 4-6초 | 4초 이하 |
