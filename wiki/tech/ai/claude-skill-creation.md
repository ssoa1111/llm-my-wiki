# Claude Skill 만들기 팁

> Claude Code Skill 문서를 단계적으로 고도화하는 8단계 프롬프트 워크플로우.

## 핵심 내용

### 8단계 Skill 작성 워크플로우

스킬 문서를 만들 때 다음 순서로 질문한다:

**1단계 — 초안 생성**
```
"[스킬 이름] 스킬의 기본 개념과 워크플로우 초안을 작성해줘."
```

**2단계 — 정량화**
```
"여기서 판단이 애매한 부분(예: '충분한', '적절한')을
체크리스트와 점수로 바꿔줘."
```

**3단계 — 분기 로직**
```
"점수나 조건이 애매한 경계선 케이스 5가지를 찾고,
각각 어떻게 처리할지 의사결정 트리를 만들어줘."
```

**4단계 — 프로그래밍 사고**
```
"이 워크플로우를 파이썬 함수처럼 표현해줘.
입력, 조건문, 반환값을 명확히 해서."
```

**5단계 — 실전 패턴**
```
"이 스킬이 처리할 대표적인 입력 사례 10개를 만들고,
공통 패턴을 추출해서 표로 정리해줘."
```

**6단계 — 메타 확장**
```
"이 스킬이 자기 자신에게도 적용된다면?
또는 이 스킬을 만드는 과정 자체를 문서에 포함한다면?"
```

**7단계 — 예시 강화**
```
"Before/After 비교 예시 3개를 만들어줘.
각각 '단순 요청 → 고도화된 결과'를 보여주는."
```

**8단계 — 최종 다듬기**
```
"이제 전체를 마크다운으로 재구성해줘.
YAML 메타데이터, 섹션 헤딩, 표, 코드블록 활용해서."
```

---

### Skill vs Tool 구분

에이전트 입장에서는 스킬이든 툴이든 모두 **도구(Tool)**로 취급한다. 에이전트는 도구의 **description(설명)**만 보고 선택한다.

```python
tools = [
    {"name": "calculator",   "description": "수학 계산을 수행합니다"},    # 툴
    {"name": "rfp_analyzer", "description": "RFP 문서를 분석합니다"},    # 스킬
]
# 에이전트는 설명만 보고 판단 — 이름의 차이일 뿐
```

**관례적 구분**:
- **Tool**: 단순 작업 (검색, 계산, API 호출)
- **Skill**: 복잡한 업무 (분석, 작성, 비교)

**핵심**: description이 명확할수록 에이전트가 올바르게 선택한다.

---

### 좋은 Skill 문서의 특징

- 입력/출력이 명확히 정의됨
- 판단 기준이 수치화됨 (모호한 표현 제거)
- 엣지케이스에 대한 처리 방법 포함
- Before/After 예시로 기대 결과 명확화
- 워크플로우가 의사결정 트리 형태로 구조화됨
