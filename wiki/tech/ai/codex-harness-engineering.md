# Codex 에이전트 우선 개발 — Harness Engineering 사례

> 코드를 직접 작성하지 않고 AI 에이전트(Codex)가 전체 소프트웨어 라이프사이클을 담당하는 에이전트 우선(agent-first) 개발 방법론. OpenAI Harness 팀의 5개월 실전 경험.

## 핵심 내용

### 결과

OpenAI 엔지니어 3명이 5개월간 Codex만으로 ~100만 라인의 코드 작성, 약 1,500개의 PR 병합 (엔지니어 1인당 하루 평균 3.5 PRs). 핵심 철학: **수동으로 작성한 코드가 없다(No handwritten code)**.

### 엔지니어의 새 역할

코드를 직접 짜지 않고:
- 큰 목표를 작은 빌딩 블록으로 세분화
- 에이전트가 실행 가능한 환경·도구·추상화 제공
- PR 검토, 피드백, 에스컬레이션 결정

> 에이전트가 막힐 때 "더 열심히 하라"고 독촉하는 게 아니라, **"어떤 도구나 문서가 부족한가?"** 를 물어보는 것이 엔지니어의 일.

---

### AGENTS.md — 백과사전이 아닌 목차

거대한 `AGENTS.md` 하나를 만드는 방식의 문제:
- 컨텍스트가 희소 리소스라 중요한 제약을 놓침
- 지침이 너무 많으면 지침이 되지 않음
- 금방 낡은 규칙의 무덤이 됨

대신: **짧은 AGENTS.md(~100줄) = 목차** + 더 깊은 정보는 `docs/` 디렉토리로 연결

```
docs/
├── design-docs/
├── exec-plans/       # active / completed / tech-debt-tracker
├── generated/        # db-schema 등 자동 생성 문서
├── product-specs/
└── references/       # 외부 라이브러리 llms.txt 등
```

계획은 일급 아티팩트: 복잡한 작업은 실행 계획(exec-plan)을 리포지터리에 저장 → 에이전트가 외부 상황에 의존하지 않고 작업 가능.

---

### 에이전트의 가독성 향상

**Chrome DevTools MCP 연결**: 에이전트가 DOM 스냅샷, 스크린샷, 탐색 작업을 직접 실행 → 버그 재현·수정 검증 가능

**로컬 관측 가능성 스택**: git worktree별 격리된 앱 인스턴스 + LogQL/PromQL 쿼리 가능한 로컬 로그·메트릭
- `"서비스 시작이 800ms 이내에 완료"` 같은 직접적인 프롬프트 가능
- 한 번의 Codex 실행이 6시간 이상 자율 작업

**핵심 원칙**: 에이전트 컨텍스트에서 접근 불가능한 것(Google Docs, Slack, 사람 머릿속)은 **사실상 존재하지 않음** → 모든 지식을 리포지터리의 마크다운으로 인코딩.

---

### 아키텍처 불변 조건 — 린터로 강제

문서화만으로는 일관성 유지 불가. 대신 **기계적 강제(linter + structural tests)**:

- 비즈니스 도메인별 고정 계층 구조: `Types → Config → Repo → Service → Runtime → UI`
- 교차 관심사(인증, 텔레메트리, 기능 플래그)는 `Providers` 인터페이스 하나로만 유입
- 커스텀 린터 오류 메시지에 **수정 지침 포함** → 에이전트가 스스로 고칠 수 있음

> 에이전트는 엄격한 경계와 예측 가능한 구조 환경에서 가장 효과적.

---

### 엔트로피 관리 — 가비지 컬렉션

에이전트가 생성한 코드는 기존 패턴을 그대로 복제하므로 시간이 지나면 드리프트 발생.

초기엔 매주 금요일(20% 시간) 수동 정리 → 확장 불가.

대신 **황금 원칙(Golden Principles)을 리포지터리에 인코딩 + 정기 정리 에이전트 운영**:
1. 직접 만든 헬퍼보다 공유 유틸리티 선호
2. 경계 검증 없이 데이터 형태 추측 금지
3. 품질 등급 추적 + 대상 리팩터링 PR 자동 생성 (대부분 1분 내 검토 후 자동 병합)

> 기술 부채는 고금리 대출. 조금씩 꾸준히 갚는 게 낫다.

---

### 자율성 수준 증가

현재 에이전트가 프롬프트 하나로 수행 가능한 작업:
- 코드베이스 현재 상태 검증
- 버그 재현 + 동영상 녹화
- 수정 구현 + 검증
- PR 열기 → 피드백 응답 → 빌드 실패 수정 → 병합

단, 이러한 자율성은 **리포지터리 특정 구조와 툴링에 강하게 의존** → 일반화 불가.

---

### 병합 철학

- 수명이 짧은 PR, 최소 병합 게이트
- 테스트 불안정성 → 후속 실행으로 해결 (무기한 대기 대신)
- **수정 비용 ≪ 대기 비용** 인 에이전트 시스템에서의 합리적 트레이드오프
