# 결제 시스템

> PG사(결제 대행사) SDK로 결제창을 띄우고, 서버에서 금액을 검증하는 기본 흐름. 클라이언트에서 온 금액과 PG사에 직접 조회한 금액을 비교하는 검증이 핵심.

## 핵심 내용

### 전체 흐름

```
고객이 결제 버튼 클릭
    ↓
PG사 결제창 팝업 (카드 정보 입력)
    ↓
PG사가 결제 처리 → 결과 반환
    ↓
우리 서버에서 검증 (금액 조작 방지)
    ↓
주문 완료 처리 + DB 저장
```

- **프론트엔드**: PG사 SDK로 결제창 띄우기만 담당
- **백엔드**: 결과 받아서 검증 + DB 저장
- **검증 필수**: 클라이언트 금액 vs PG사 직접 조회 금액 비교 (해킹 방지)

---

### 금액 검증 로직

```
1. 프론트에서 받은 정보:
   - 주문번호: ORDER123
   - imp_uid: imp_123456 (PG사 결제 ID)

2. 우리 DB 확인: ORDER123 = 10,000원?

3. PG사 API 직접 조회:
   GET /payments/imp_123456 → 실제 결제 금액 확인

4. 비교:
   프론트 금액 == DB 금액 == PG 금액 → 성공
   불일치 → 즉시 결제 취소 + 로그
```

---

### 불일치/중복 처리

| 케이스 | 처리 방법 |
|--------|---------|
| 금액 불일치 | 결제 즉시 취소 + "사기 시도" 표시 + 알림 |
| 이미 처리된 주문 (중복) | 두 번째 결제 취소 |
| DB에 없는 주문 (위조) | 결제 취소 + 차단 |

**중복 결제 방지**:

```python
if 주문이_이미_완료_상태:
    새_결제_취소()
    return "이미 처리된 주문입니다"

# 또는: 주문번호에 타임스탬프 포함 → 물리적으로 중복 불가
merchant_uid = f"ORDER_{timestamp}_{user_id}"
```

중복 결제가 발생하는 흔한 케이스: 네트워크 지연으로 사용자가 다시 결제, 뒤로가기 후 재결제, 개발자 도구로 중복 요청 시도.

---

### PG사 선택

- **토스페이먼츠**: 국내 최대, 문서 좋음
- **아임포트(포트원)**: 다양한 PG사 통합 지원
- **나이스페이**: 국내 전통 PG사

직접 카드사와 연동하는 경우는 거의 없음 (복잡도와 비용 때문).

---

### 구현 순서

1. PG사 회원가입 + API 키 발급
2. SDK 설치 (`npm install`)
3. 결제 요청 API 호출 (결제창 팝업)
4. 콜백으로 imp_uid 수신
5. 서버에서 PG사 API로 직접 검증
6. 주문 정보 DB 저장
